# Phase 1 — Mountains (Vergeo5, zero → production)

**Date:** 2026-07-06 · **Inputs:** `00-decisions.md` (locked) + `research/*` + `docs/designs/SELECTION.md` (tokens) + `docs/ops/lenco/lenco-api-distilled.md` (payment contracts). 16 mountains. Priorities: **P0** blocks launch · **P1** launch week · **P2** post-launch. Pebble counts are estimates for Phase 2.

---

## M01 — Foundations & Infrastructure — **P0**
**Scope:** Monorepo (pnpm + turborepo: `apps/customer|vendor|admin`, `services/api`, `packages/ui|types|config|i18n`), FastAPI service skeleton (health, settings, error envelope, OpenAPI), Supabase project wiring (migrations pipeline via CLI, generated types), OCI VM provisioning (Docker Compose: api, caddy, n8n), Caddyfile + TLS + Cloudflare DNS, Vercel project for customer app, GitHub Actions CI (ruff/mypy/pytest; eslint/tsc/vitest; build; deploy jobs), environments (local/staging/prod) + secrets convention, nightly DB dump → OCI Object Storage, rollback runbook.
**Why critical:** every wave lands on these rails; parallel Cursor agents need deterministic commands and CI gates from day 1.
**Success criteria:** fresh clone → `pnpm i && make dev` runs all apps against local/staging Supabase; CI green on a hello-world slice through all apps; deploy + rollback demonstrated on staging; secrets never in repo (CI scanner proves it).
**Dependencies:** none. **Risk:** M — biggest: solo-ops OCI snowflake config → everything in compose/Caddyfile committed, no console-clicking. **Pebbles:** ~7.

## M02 — Design System & UI Kit — **P0**
**Scope:** Token set from `docs/designs/SELECTION.md` (colors incl. cream ground/navy serif display/aubergine panels/pastel category fills, type scale, spacing, radii, shadows, motion durations) as Tailwind theme in `packages/ui`; core components: buttons, inputs/selects/OTP field, cards (product/event/service/vendor), badges + corner ribbons, pills/tags, modals/sheets, toasts, skeletons, tabs, bottom mobile nav + top nav, price/K-formatting primitives, star-rating, stepper; loading/empty/error patterns; a11y (focus, contrast, touch targets ≥44px); zero hardcoded user-facing strings (i18n keys only); component preview page.
**Why critical:** exclusive-file-ownership waves can only stay conflict-free if shared UI is finished first; visual consistency across 3 apps.
**Success criteria:** all v1 screens buildable from kit; preview page renders every component in EN with i18n keys; Lighthouse a11y ≥95 on preview; tokens change in ONE file propagates everywhere.
**Dependencies:** M01; SELECTION.md (done). **Risk:** M — biggest: 6 design files still missing → tokens locked from committed variants; late files can only add variants to the merchandising library, not retoken the system. **Pebbles:** ~8.

## M03 — Data Core (Schema, RLS, Types, Search Projection) — **P0**
**Scope:** Full Postgres migration set: profiles/roles, vendors + locations + KYC records (tiered), category tree, canonical `products`, `vendor_listings` (condition, stock modes, `wholesale`+`price_tiers`+`moq`), `services`, `events`/`event_instances`/`ticket_types`/`tickets`, orders spine (`orders`/`order_items` + `item_kind` detail tables), escrow ledger (double-entry postings) + payouts, reviews, disputes, returns, config tables (commissions/fees/zones/quotas/flags/merch slots), notification outbox, `search_documents` projection + sync triggers, pgvector columns + indexes; **RLS policies on every table** (service-role bypass only server-side); seed framework; TS type generation into `packages/types`; ERD doc.
**Why critical:** the single most contention-prone artifact — must be merged before any parallel wave; encodes D24's model decision.
**Success criteria:** `supabase db reset` builds clean; RLS test suite proves customer/vendor/admin isolation (incl. cross-vendor denial); seed produces a browsable demo dataset; types compile in all apps; EXPLAIN on hot queries (PLP, search, comparison) uses indexes.
**Dependencies:** M01. **Risk:** H — biggest: schema churn after waves start → comparison view, escrow ledger, and ticket tables reviewed against Bible briefs before merge; additive-only migrations rule afterward. **Pebbles:** ~10.

## M04 — Auth & Accounts — **P0**
**Scope:** Supabase Auth: phone-OTP primary (Africa's Talking SMS hook), email+password, Google OAuth; profile bootstrap on signup; role claims (customer/vendor/admin) in JWT + RLS integration; middleware/guards in all 3 apps + FastAPI dependency; account pages (profile, addresses w/ landmark + GPS pin, language pref, notification prefs, delete/export per DPA); rate limiting + OTP abuse guards (per-number/IP caps, cooldowns); session security (refresh rotation).
**Why critical:** everything user-scoped hangs off identity; OTP is the mass-market door (no-email users).
**Success criteria:** signup→browse→order on a phone number only; role escalation impossible from client (tested); OTP brute-force blocked (tested); account deletion cascades correctly.
**Dependencies:** M01, M03. **Risk:** M — biggest: SMS OTP cost/deliverability → email/WhatsApp-OTP fallback path + resend throttling. **Pebbles:** ~7.

## M05 — Catalog, Search & Discovery — **P0**
**Scope:** Customer home (merchandising slots driven by M13 config: hero variants, banners, featured collections, events row first per IA); category browse + PLP (facets: category/price/location/rating/availability/condition; sort incl. nearest/cheapest); PDP (gallery ≤8 imgs, variants, specs, vendor block, **multi-vendor comparison** "N vendors selling this"); supplies tab (tier-price display, MOQ); **directory tab** (vendor cards, search/filter, profile pages); hybrid search — Postgres FTS + pg_trgm typo tolerance + pgvector semantic, RRF fusion, autocomplete, Bemba/Nyanja term aliases; Cloudinary pipeline (f_auto/q_auto, responsive srcset, LQIP, lazy); SEO: schema.org Product/Offer/AggregateRating/LocalBusiness/Event, sitemaps, canonicals, OG images.
**Why critical:** discovery IS the product thesis ("commerce-powered discovery platform"); SEO is the free acquisition channel.
**Success criteria:** search returns relevant results for exact ("itel A70"), fuzzy ("chitange"), and semantic ("dress for kitchen party") queries on seed data; comparison view sorts by price/distance; PLP LCP ≤2.5s on Fast-3G lab profile; rich results validate in Google testing tool.
**Dependencies:** M02, M03. **Risk:** M — biggest: relevance on thin launch data → curated boosts (in-stock, below-median price, verified vendor) + synonym table from day 1. **Pebbles:** ~11.

## M06 — AI Mode "Ask Vergeo" — **P1**
**Scope:** Embedding pipeline on publish/update (OpenRouter cheap model or edge gte-small; batch + retry; `search_documents`-fed); "Ask Vergeo" tab + inline search entry: conversational RAG grounded ONLY in platform listings (products/services/events/supplies/vendors), structured filter extraction (price/category/location), answers as text + cited listing cards, ZMW-correct; quotas (guest 3 lifetime → signup; free 25/mo; config-table), response cache, per-answer token caps, **$15/mo global kill-switch**, abuse filters; analytics on queries (zero-result mining).
**Why critical:** founder-differentiating feature (Alibaba-style AI tab); also mines demand signal for merchandising.
**Success criteria:** 20-question eval set answers grounded with zero fabricated listings; quota + kill-switch enforced (tested); p95 answer <6s; cost per answer ≤$0.002 at default model.
**Dependencies:** M05. **Risk:** M — biggest: hallucination/cost drift → strict grounding prompt + evals in CI + caps. **Pebbles:** ~6.

## M07 — Cart & Checkout — **P0**
**Scope:** Cart (guest + authed, merge on login, per-vendor grouping, stock/price revalidation); checkout steps: contact/OTP-if-guest → delivery (Lusaka zones + fees, free ≥K200) or pickup point/vendor location → payment method (MoMo rail select w/ number, card, COD ≤K500 gate) → review/confirm; landmark + GPS-pin address capture; 10–15min stock reservation with countdown; atomic order creation (multi-vendor split into sub-orders); order-pending screen driving M08 USSD-push wait UX; abandoned-checkout event emission (n8n later).
**Why critical:** the conversion funnel; every kwacha passes through it.
**Success criteria:** two concurrent buyers cannot oversell last unit (race test); guest→auth merge preserves cart; checkout completes in ≤4 steps on 360px; reservation expiry restocks (tested); COD gate enforces ≤K500.
**Dependencies:** M03, M04, M05. **Risk:** H — biggest: oversell/race conditions → DB reservation w/ row locks + property tests. **Pebbles:** ~8.

## M08 — Payments & Escrow (Lenco) — **P0**
**Scope:** Payment abstraction (strategy pattern; Lenco first, seams for Flutterwave/PawaPay); Lenco integration per `docs/ops/lenco/lenco-api-distilled.md`: MoMo collections USSD-push via direct API (**MTN/Airtel; Zamtel collections unconfirmed — F9a; Zamtel payouts supported**) with `pay-offline` status webhook + mandatory reconciliation poller, **card via Lenco hosted widget only** (no PCI scope; server-side verify before fulfilment), payouts (MoMo ≈instant incl. Zamtel, bank 24–36h) with pre-payout `/resolve` name-match, **refunds modeled as payouts (no Lenco refunds API)**; webhook endpoint (signature verification, replay protection, idempotency keys end-to-end); **double-entry escrow ledger** postings for every money event; release rules (buyer confirm / 48h auto after delivered / 7d after shipped; dispute holds); payout execution + batching + failure retry; refunds (full lane-1 / computed lane-2) from escrow + post-release clawback; COD reconciliation entries; commission calculation from config; daily reconciliation report (Lenco balance vs ledger); ZRA-sequential invoice/receipt numbering hooks; money = integer ngwee everywhere.
**Why critical:** the money. Escrow trust is the moat; a ledger bug is existential.
**Success criteria:** sandbox E2E: charge → webhook → escrow hold → delivery confirm → release → payout, for all 3 rails + card; ledger always balances (property test: sum of postings = 0); duplicate/out-of-order/forged webhooks handled (tested); reconciliation report matches sandbox to the ngwee; every payment endpoint authz-checked + rate-limited.
**Dependencies:** M03, M07. **Risk:** **H (highest in project)** — biggest: USSD-push edge cases (timeout, user-cancel, phone off) leaving orphaned states → explicit pending-state machine with expiry sweeper + customer retry UX + idempotent ledger. **Pebbles:** ~12.

## M09 — Orders, Fulfilment & Post-Order — **P0**
**Scope:** Order state machine (Placed→Confirmed→Processing→Ready/Shipped→Delivered→Completed; Cancelled branches) with guarded transitions + audit log; vendor actions (confirm/reject w/ reason, pack, ship w/ tracking note, ready-for-pickup); **QR+PIN pickup** (issue on ready, vendor scanner verify, single-use); manual-dispatch flow surfaces (admin books Yango/inDrive/courier, pastes tracking; customer sees timeline); customer order pages (timeline, receipt/invoice download, confirm-received, report-problem); returns lane 1 (faulty ≤48h, evidence upload) + lane 2 (vendor-opt-in, fee computation) → refund requests into M08; disputes (evidence, vendor response, admin resolution); auto-triggers (48h auto-confirm, 7d auto-release) via scheduled jobs.
**Why critical:** operational truth of the marketplace; trust promises (escrow release timing) live here.
**Success criteria:** every legal transition tested, every illegal one rejected; QR verify offline-tolerant + single-use (tested); auto-confirm/release jobs idempotent under re-run; refund math (lane 2: price − both-way delivery − 10%) exact in tests.
**Dependencies:** M03, M07, M08. **Risk:** H — biggest: state × refund × dispute matrix correctness → exhaustive transition test table generated from spec. **Pebbles:** ~10.

## M10 — Events & Ticketing — **P0**
**Scope:** Organiser flow (vendor-KYC reuse + organiser flag): event CRUD (venue, date/instances, capacity, images), ticket types (fixed, multi-tier, free RSVP; caps per D2); listing pages + events home row + calendar-lite filter ("Tonight/This Weekend" default); purchase through M07/M08 (5% commission, 0% free events); **ticket wallet**: dynamic QR (HMAC over 60s windows) + 6-digit PIN backup, offline-viewable; **scanner mode** (organiser PWA: camera scan, offline verification cache, first-scan-wins, live count); simple transfer-to-friend (until T-6h); organiser dashboard-lite (sales, check-ins); event escrow timing (T+24h post-event ≤14d out; else 50/50 T-7/T+1) per D5; Event schema.org SEO.
**Why critical:** headline v1 vertical (deck leads with events); dynamic-QR trust is a visible differentiator vs paper/WhatsApp tickets.
**Success criteria:** full cycle on sandbox: create → publish → sell (paid+free) → QR at door validates once and only once (offline included) → escrow settles per rule; screenshot of QR fails verification after 60s window (tested); oversell impossible at capacity boundary (race test).
**Dependencies:** M02, M07, M08 (M09 for states). **Risk:** M-H — biggest: offline scanner correctness (clock skew, cache staleness) → signed time-window tolerance ±1 window + sync-on-reconnect tests. **Pebbles:** ~9.

## M11 — Services & RFQ — **P1**
**Scope:** Service listings (portfolio gallery, service area, from-price optional), provider profiles w/ response-time badge; **post-a-job**: need description + category + preferred date + budget band → broadcast to matching providers (cap ~8); quote inbox/compare; accept → deposit (12% commission basis) via escrow; balance on completion; completion confirm + review; light moderation (contact-info stripping pre-acceptance).
**Why critical:** completes "everything Zambia offers" thesis; near-pure reuse of M05/M08/M09 machinery (cheap to add, high narrative value).
**Success criteria:** job→quotes→accept→deposit→complete→balance→review E2E on sandbox; providers can't see each other's quotes; pre-acceptance messages strip phone/WhatsApp patterns (tested).
**Dependencies:** M05, M08, M09. **Risk:** M — biggest: liquidity (few providers at launch) — ops problem; product keeps posting friction near-zero. **Pebbles:** ~6.

## M12 — Vendor Portal — **P0**
**Scope:** Vendor onboarding: apply → **T1 KYC** (NRC photo + selfie + MoMo-name match, same-day) → **T2** (PACRA + company TPIN); status machine + caps enforcement (30 listings; first-5-orders ≤K500; payout velocity); listing management: **search-and-attach to canonical**, submit-new-canonical (→ M13 moderation), commodity quick-list, **CSV bulk import** (validate/preview/apply), image upload pipeline, stock + price (+ tier prices for supplies), condition + evidence fields; orders queue (mobile-first daily-driver: today's takings, needs-action, big buttons); payouts view + statements (ledger-backed); storefront profile (logo, hours, location pin, description → doubles as directory entry); basic analytics (sales, orders, views); vendor pitch/landing page (public, recruits vendors — D10).
**Why critical:** supply side IS the cold-start battle; founder recruits with this portal in hand.
**Success criteria:** unassisted T1 vendor: signup → KYC → first listing live ≤10min (usability-tested with seed persona); CSV import of 100 rows with mixed errors gives row-level feedback; caps enforced server-side (tested); daily-driver usable one-handed at 360px.
**Dependencies:** M02, M03, M04. **Risk:** M-H — biggest: onboarding friction kills recruitment → founder dry-runs the flow before beta; demo sandbox vendor for pitches.
**Pebbles:** ~11.

## M13 — Admin App & Merchandising — **P0**
**Scope:** Hardened admin (separate origin/subdomain, admin-role RLS, IP allowlist/Cloudflare Access, audit log on every mutation); queues: KYC review (docs viewer, approve/reject w/ reasons), canonical-product moderation (dupe detection), listing/review flags; disputes console (evidence, decisions → M08 refund actions); order ops (search any order, manual dispatch console, status intervention, escrow manual hold/release with dual-note); **config editors**: commissions, delivery zones/fees, COD cap, quotas, feature flags, category tree; **merchandising manager**: hero variant library (from design variants), banner slots, featured-collection builder, schedule/rotation — the founder's admin-swappable elements; dashboards: GMV, orders, payout liabilities, reconciliation status, vendor/product counts, AI-mode usage + cost; support inbox-lite (order lookup + canned WhatsApp/SMS replies); n8n hooks for daily digest.
**Why critical:** solo-founder force multiplier; escrow/refund powers and moderation keep the platform safe and legal.
**Success criteria:** every admin mutation audited (who/what/before/after); merch change (swap hero, reorder collections) reflects on customer home ≤1min without deploy; KYC queue processes a submission end-to-end; reconciliation dashboard flags an injected ledger mismatch.
**Dependencies:** M03, M04, M08. **Risk:** M — biggest: admin scope creep → everything beyond these queues/config/dashboards goes to n8n or Phase 2. **Pebbles:** ~11.

## M14 — Notifications & Automation — **P0**
**Scope:** Notification outbox (DB table, at-least-once dispatcher, per-channel adapters, dedupe keys); **WhatsApp Cloud API** client (templates per `docs/ops/whatsapp-cloud-api-setup.md`: order_confirmed, payment_received, order_shipped/ready, order_delivered, vendor_new_order, otp_login; webhook for status; quality-safe opt-in/STOP handling); SMS fallback (Africa's Talking) on WhatsApp failure/no-opt-in; email tertiary (Resend: receipts, KYC outcomes); per-user channel preferences; n8n workflows: vendor KYC nudge, daily ops digest to founder, payout-failure alert, low-stock alert, review request post-completion, abandoned-cart (flag-gated for later); template i18n (EN now, Bemba/Nyanja slots).
**Why critical:** WhatsApp-native touchpoints are a core market requirement; notifications carry the trust narrative ("your money is held safely").
**Success criteria:** order lifecycle fires correct messages exactly once per event (dedupe tested); WhatsApp failure → SMS within 2min (tested with forced failure); STOP honored across channels; founder digest arrives daily with real numbers.
**Dependencies:** M03, M08, M09. **Risk:** M — biggest: Meta template approval/verification delays → templates submitted at F5 time, SMS covers any gap. **Pebbles:** ~7.

## M15 — Trust, Security, Compliance & Legal — **P0**
**Scope:** Reviews (verified-purchase only, 1–5★ + text + photos, vendor replies, report/moderation, Bayesian aggregate on cards); platform security hardening pass: security headers/CSP, rate-limit sweep on all mutating + auth endpoints, input-validation fuzz pass, OWASP Top-10 checklist audit vs codebase, dependency + secret scanning in CI, pen-test-lite script; legal pages: T&Cs, Privacy (Zambia Data Protection Act: consent, retention, export/delete), Returns/Refunds policy (D17 lanes), Vendor Agreement (commissions, payout terms, prohibited items); **ZRA readiness**: sequential tax-invoice/receipt numbering, VAT-flag per category (off at launch), VSDC API seam documented + stubbed; compliance config (prohibited-category enforcement per D8); backup restore drill + disaster-recovery runbook.
**Why critical:** "production ready" definition demands it; escrow + KYC docs + payments make this a regulated-adjacent system with real PII.
**Success criteria:** OWASP checklist zero criticals; restore drill recovers staging from backup ≤30min; legal pages linked from all footers + checkout consent; invoice numbers gapless-sequential under concurrency (tested); a prohibited-category listing attempt is blocked server-side.
**Dependencies:** spans M03–M14 (final pass), starts early. **Risk:** M-H — biggest: escrow legal structure under NPS Act 2026 (F4, external counsel) → launch gate, not build blocker. **Pebbles:** ~9.

## M16 — Performance, PWA, Content, Analytics & Launch QA — **P0**
**Scope:** Performance budget enforcement in CI (bundle-size guards ≤150KB gz customer routes; image lint; Lighthouse CI vs Fast-3G/360px: LCP ≤2.5s, Perf ≥90/SEO ≥95/A11y ≥95); PWA (installable, offline shell + cached catalog browsing tolerance, install prompt UX); i18n completeness lint (no hardcoded strings, locale pseudo-test) + ZMW/date formatting audit; content pages (about, contact, FAQ/help center seed, vendor pitch page polish, 404/500); analytics (GA4 + server-side event log: search terms, zero-results, funnel steps, AI-mode usage); observability: Sentry (API+web), UptimeRobot, structured logs, error-budget doc; **E2E suite** (Playwright): browse→search→PDP→cart→checkout→Lenco-sandbox pay→order confirm→WhatsApp-mock; vendor list→sell; event buy→scan; k6 load test (100 concurrent checkout); beta program tooling (invite gate, feedback widget); launch checklist + go/no-go doc.
**Why critical:** this is the "production ready" gate itself — 3G reality, observability, and proof the critical paths work.
**Success criteria:** all budgets green in CI on launch commit; E2E suite green on staging against Lenco sandbox; load test p95 <500ms at 100cc; beta cohort can be invited/gated; go/no-go checklist signed.
**Dependencies:** all. **Risk:** M — biggest: perf regressions from parallel waves → budgets enforced per-PR from Wave 1, not retrofitted. **Pebbles:** ~9.
