## 1. 🔴 CRITICAL — Refund double-execution: non-atomic guard + ignored idempotency key => double payout & double escrow drain

- **Dimension:** money-ledger-escrow · **Confidence:** high
- **Location:** `services/api/app/services/refunds/execute.py:238`
- **Problem:** execute_refund's only dedup is _find_existing_refund(order_id) (a plain SELECT on refunds by order_id) followed by an insert — a classic TOCTOU with no serialization. The refunds table has no unique constraint on order_id (0006_money.sql:113 is a plain non-unique index refunds_order_id_idx), so two concurrent/retried executions both pass the check, both insert distinct refund rows, and both post ledger transactions. The ledger idempotency keys are refund-{refund_id}-ledger / -clawback keyed on a freshly generated per-call refund_id (lines 183-204, 215-222), so post_transaction's idempotency cannot collapse them. The request/caller idempotency_key (e.g. dispute-{dispute_id}-refund) is only written into breakdown (lines 283-284) and is never passed to post_transaction — the dispute comment 'all M08 calls are idempotency-keyed, so a retry can never double-pay' (disputes/service.py:345) is therefore false.
- **Failure scenario:** Admin double-submits POST /refunds/execute for order O (or dispute auto-resolve + manual admin fire, or a retried request handled concurrently in FastAPI's threadpool). Both calls run _find_existing_refund(O) before either inserts -> both see no active refund -> both insert refund rows R1,R2 -> both post REFUND_LANE1/2 (or CLAWBACK) ledger transactions draining escrow twice for the same order -> both call initiate_customer_refund_payout creating two pending Lenco payouts. Customer is refunded twice and the ledger over-releases escrow by one full refund amount.
- **Verifier trace:** Traced execute_refund (execute.py:226-353) end to end. Dedup is solely _find_existing_refund (execute.py:99-117): a plain SELECT on refunds by order_id with a status-in-{pending,processing,completed} check, no FOR UPDATE / advisory lock, followed by a plain supabase .insert() (line 294) with no ON CONFLICT. Confirmed 0006_money.sql:113 is a non-unique index (refunds_order_id_idx); the table PK is id only, so the DB accepts a second row for the same order_id. Confirmed refund_id is minted fresh per call via uuid.uuid4() (line 273) and the ledger idempotency keys are refund-{refund_id}-ledger/-clawback (lines 184/194/216); post_transaction dedups via ON CONFLICT(idempotency_key) on ledger_transactions (engine.py:236), so two concurrent calls with two distinct refund_ids produce two distinct keys and both REFUND/CLAWBACK transactions post, draining escrow twice. Confirmed the caller idempotency_key (e.g. dispute-{id}-refund) is only written to breakdown (lines 283-284) and never forwarded to post_transaction, so the disputes/service.py:345 comment is false. payout_port.initiate_customer_refund_payout (payout_port.py:54-71) likewise mints a fresh payout_id + refund-id-derived reference per call with no dedup, yielding two pending customer payouts. middleware.py provides only X-Request-ID, no request-level idempotency. Nuance: the endpoint is async def calling blocking sync code, so within one event-loop worker calls serialize and pure sequential post-completion retries are caught by the status guard; but production multi-worker deployments and the two distinct trigger paths (dispute auto-resolve vs manual admin execute) provide a real concurrency window. This is a genuine TOCTOU double-refund / double-escrow-drain on escrow money — critical.

## 2. 🔴 CRITICAL — Concurrent checkout submit creates duplicate orders (idempotency check outside the lock, no in-transaction status recheck, no unique constraint)

- **Dimension:** checkout-stock-races · **Confidence:** high
- **Location:** `services/api/app/services/orders/create.py:512`
- **Problem:** create_orders_atomic performs all idempotency/status guards via PostgREST reads BEFORE the transaction: _fetch_existing_by_idempotency_key (line 417) matches on the client idempotency_key, but the checkout_groups row still holds its server-generated 'chk-*' key until this call overwrites it on completion (line 588-589), so a first-time submit never matches; _fetch_checkout_session (line 423) and _session_expired (line 441) both read status='pending'. Inside the transaction the only synchronization is `SELECT ... FOR UPDATE` on the checkout_groups row (lines 512-514) with NO `status='pending'` predicate and no re-read of status. The blind INSERTs into public.orders use freshly generated uuid4 order_ids each call, and orders has no unique constraint on (checkout_group_id, vendor_id) (verified in 0005_orders.sql and all later migrations). So two concurrent requests both pass the pre-checks, serialize on the row lock, and each insert a complete set of orders for the same checkout_group_id.
- **Failure scenario:** Customer double-clicks 'Place order' (or client retries) firing two POST /orders with the same session_id and same idempotency_key while both are in flight. Both pre-transaction idempotency reads see the session still keyed 'chk-*' and status 'pending' -> both proceed. T1 takes the FOR UPDATE lock, inserts orders (ids A) for vendor V, deletes reservations, sets status='completed', idempotency_key=client_key, commits. T2 then acquires the lock, does NOT re-check status, inserts a SECOND set of orders (ids B) for vendor V, deletes 0 reservations, re-sets the same key (same row -> no unique violation), commits. Result: two orders per vendor for a single checkout_group with a single payment -> vendor fulfils/ships twice for one payment, or customer is double-billed if payment is initiated per order. The client-supplied idempotency_key provides no protection under true concurrency.
- **Verifier trace:** Traced the full concurrent path and confirmed every claim.

1. Session key is 'chk-*' until completion. checkout.py:463 creates the checkout_groups row with idempotency_key = f"chk-{secrets.token_urlsafe(16)}", status 'pending'. The client idempotency_key is only written onto the row at completion (create.py:588-589). So on a first-time submit, _fetch_existing_by_idempotency_key (create.py:417, matching eq("idempotency_key", client_key)) finds no row -> None -> guard at line 420 does not fire. CONFIRMED.

2. Pre-transaction guards are PostgREST reads (TOCTOU). _fetch_checkout_session (line 423) and _session_expired (line 441) read status='pending'/reservation expiry via the REST client, outside any lock. The router (orders_create.py) adds only _ensure_session_active (another PostgREST read) and no distributed lock/idempotency table. CONFIRMED.

3. Each request is its own DB transaction. run_sql_script (claim.py:44-61) spawns a fresh psql subprocess per call with its own BEGIN;...COMMIT; block on a separate connection. So two concurrent create_orders_atomic calls run as two independent transactions. CONFIRMED.

4. In-transaction sync is a row lock with NO status recheck. The only synchronization is SELECT id::text FROM public.checkout_groups WHERE id=session AND customer_id=customer FOR UPDATE (create.py:510-516). It selects only id, has no status='pending' predicate, and the result's status is never inspected. After serializing on the lock, the code unconditionally proceeds to INSERT orders. CONFIRMED.

5. orders have fresh uuid4 ids and NO unique constraint on (checkout_group_id, vendor_id). order_id/order_item_id are uuid4() per call (create.py:476,479). Reviewed 0005_orders.sql: orders has only a PK on id plus non-unique indexes (orders_checkout_group_id_idx etc.); grep across all migrations found no unique index on orders(checkout_group_id,vendor_id) (only stock_reservations and funnel_events carry checkout_group uniqueness). CONFIRMED.

Reproduced scenario: T1 and T2 both submit with same session_id + same client idempotency_key while in flight. Both pass the pre-transaction reads (session still 'chk-*' keyed, status 'pending') before either commits. T1 takes FOR UPDATE, inserts orders (ids A), deletes reservations, sets status='completed'/idempotency_key=client_key, commits, releases lock. T2 then acquires the lock, does NOT recheck status, inserts a SECOND full set of orders (ids B) for the same checkout_group_id/vendor, its DELETE reservations affects 0 rows, and its UPDATE sets the SAME row's idempotency_key to the SAME value (no unique violation) and commits. The duplicate-key replay net at line 607 never fires because no unique constraint is violated. Result: two order sets per vendor for one checkout_group / one payment -> double fulfilment or double billing.

The only mitigating factor is the FOR UPDATE serialization, but because there is no in-transaction status/existence recheck and no unique constraint, it fails to prevent the duplicate — it merely orders the two duplicating writes. Finding is real and correctly rated critical (financial/fulfilment integrity).

## 3. 🟡 MAJOR — Invoice numbering is not gapless: number allocation and invoice-row insert run in separate DB transactions

- **Dimension:** money-ledger-escrow · **Confidence:** high
- **Location:** `services/api/app/services/invoicing/builder.py:175`
- **Problem:** issue_receipt/issue_tax_invoice call allocate_invoice_number(series) and then _persist_invoice(payload) as two independent operations. allocate_invoice_number (allocation.py:19) runs SELECT public.next_invoice_no(series) via run_sql_script, which spawns a standalone psql process (orders/audit.py:32-49) — a separate connection that autocommits. next_invoice_no (0006_money.sql:217) increments and COMMITS invoice_counters.next_no inside that call. _persist_invoice then runs the INSERT into public.invoices in a _different_ psql process/transaction. There is no shared transaction spanning allocation and persistence, so a failure/crash/exception at or before persistence permanently burns the allocated number with no invoice row — a gap. This contradicts the documented 'gapless' guarantee (comments at 0006_money.sql:161-162,248 and builder docstrings) required for ZRA-ready sequential invoices.
- **Failure scenario:** issue_tax_invoice allocates TAX #500 (counter committed to 501), then _persist_invoice fails (DB error, constraint hiccup, or process restart between the two psql calls). No invoices row with no=500 is ever written; the next successful issue produces #501. The TAX series now has a permanent gap at 500, breaking the gapless/ZRA sequential-invoice compliance requirement (and any audit that reconciles the series).
- **Verifier trace:** Traced builder.py:197,207 → allocation.py:19 → audit.py:32-49 → 0006_money.sql next_invoice_no. Confirmed: issue_tax_invoice/issue_receipt perform allocation and persistence as two independent run_sql_script calls, each of which spawns a separate `psql` subprocess (audit.py:33: subprocess.run(["psql", ...])) = separate connection with autocommit. next_invoice_no does SELECT ... FOR UPDATE then UPDATE next_no = next_no + 1; the FOR UPDATE lock is held only for that single psql invocation, and the increment is durably committed by that first subprocess regardless of downstream success. _persist_invoice then runs a second, distinct psql subprocess to INSERT into public.invoices, and raises RuntimeError if it fails. There is no transaction spanning both operations — architecturally there cannot be, since they are two separate processes. So if persistence fails (order_id FK violation, connection drop, ON_ERROR_STOP abort, or process restart between calls), the counter has advanced (e.g. 500→501) with no invoices row at 500; the next issue yields 501, leaving a permanent gap. Skeptic checks all fail to save it: grep shows only test callers (no shared-transaction wrapper), no retry reuses the allocated number (retry allocates a fresh one, widening the gap), no compensating logic exists, and the unique(series,no) constraint + FOR UPDATE only guard duplicates/concurrency, not the burned-number gap. This contradicts the documented gapless guarantee (migration comment 'Gapless invoice numbering' and builder docstrings 'gapless sequential number') for ZRA-ready sequential invoices. Major is appropriate: real design defect, but latent (only triggers on a failure between the two calls) and fiscalisation is not yet wired (zra-readiness.md).

## 4. 🟡 MAJOR — Order creation TOCTOU vs stock sweeper -> oversell (stock never re-verified inside the atomic transaction)

- **Dimension:** checkout-stock-races · **Confidence:** high
- **Location:** `services/api/app/services/orders/create.py:584`
- **Problem:** Expiry/hold validity is checked only by _session_expired (line 441) via a separate PostgREST read, after which several more round-trips run (_load_commission_rates, _load_listing_commission_keys, _load_product_ids at lines 461-464) before the order transaction executes at line 603. The transaction consumes the hold with a bare `DELETE FROM stock_reservations WHERE checkout_group_id=...` (_consume_reservations_sql, line 99-102, invoked line 584) that does not verify a row was actually deleted and never re-decrements or re-checks vendor_listings.stock_qty. Meanwhile sweep_expired_reservations() -> release_reservation() deletes the reservation unconditionally by (listing_id, checkout_group_id) and adds qty back to stock_qty (release.py lines 36-47), re-checking neither expires_at nor order state.
- **Failure scenario:** A reservation is a second from expiry when the customer submits. _session_expired reads expires_at>now -> not expired, so the flow proceeds; the multi-round-trip commission/product loads (hundreds of ms to seconds) push past expires_at. The internal stock sweeper runs in that window: release_reservation DELETEs the (now-expired) reservation and does stock_qty += qty, returning the units to available inventory. The order transaction then commits: it DELETEs 0 reservations (already gone), creates the order, marks the session completed. Net effect: the customer holds a real order but the reserved units were handed back to inventory and can be claimed/sold to another customer -> oversell; the vendor owes more units than exist. The atomic order section never re-asserts the hold or re-decrements stock, so nothing catches the released hold.
- **Verifier trace:** Traced the full path. Stock is decremented at claim (claim.py:147-152); a reservation row represents held/decremented units. Order consumption (_consume_reservations_sql, create.py:99-102, invoked line 584) is a bare `DELETE FROM stock_reservations WHERE checkout_group_id=...` with no row-count check and no stock re-decrement/re-assertion. The sweeper (internal_stock_sweeper.py cron → sweep_expired_reservations → release_reservation, release.py:34-48) selects expires_at<now, DELETEs the reservation, and does stock_qty += qty, re-checking neither expiry-at-delete nor order state.

The TOCTOU is real and unguarded: _session_expired (create.py:441) is a separate read taken before _load_commission_rates/_load_listing_commission_keys/_load_product_ids (461-464) and before the transaction opens at line 603, so a reservation valid at check time can expire during those round-trips. Crucially, no lock serializes the order tx against the sweeper: the order tx locks only checkout_groups FOR UPDATE (512-514), but release_reservation never touches checkout_groups — only stock_reservations and vendor_listings. If the sweeper commits its release (restock) before the order tx's DELETE runs (the pre-BEGIN loads at 461-464 provide the window), the order tx deletes 0 rows and completes anyway: it inserts the order and marks the session completed while the units have been handed back to vendor_listings.stock_qty → oversell.

No safety net catches it: revalidate_lines is used only in vendor_listings_manage.py, not the order path (orders_create.py:249 calls create_orders_atomic with no re-claim/stock recheck); the only stock constraint is `stock_qty >= 0` (0003_catalog.sql:95) which prevents negative stock, not oversell-via-restock; no trigger on orders re-validates reservations. Severity major (not critical) because it needs the cron tick to coincide with a reservation crossing expiry mid-submission — a narrow but real window — with genuine oversell impact.

## 5. 🟡 MAJOR — Payment retry after failure/expiry is impossible — Lenco reference collides on the UNIQUE constraint

- **Dimension:** webhooks-state-machines · **Confidence:** high
- **Location:** `services/api/app/services/payments/initiate.py:94`
- **Problem:** lenco_reference is derived deterministically from a stable id: reference_source = request.order_id or request.checkout_group_id, then make_order_reference() base64-encodes it (no per-attempt salt). payments.lenco_reference carries a UNIQUE constraint (migration 0006_money.sql:45). The card path does the same (payments_card.py:498). Meanwhile the payment state machine explicitly builds a retry flow: sweep_stale_payments/reconcile call release_checkout_for_retry (state.py:406-430) which resets checkout_groups.status to 'pending', and initiate only blocks when status=='completed' (initiate.py:80). So a second attempt on the same checkout group is permitted by every guard except the DB, where the INSERT re-uses the identical lenco_reference and violates payments_lenco_reference_key. The raised postgrest APIError is not handled (the check at initiate.py:107 only catches an empty result, not an exception), so it surfaces as a 500.
- **Failure scenario:** Customer's first MoMo/card payment for checkout group G fails or times out. The sweeper transitions it to failed/expired and calls release_checkout_for_retry(G) -> checkout_groups.status='pending'. Customer taps 'retry payment'. initiate_checkout_payment(G) computes lenco_reference = ord-<b64(G)> (identical to the first payment's) and INSERTs a new payments row -> UNIQUE(lenco_reference) violation -> 500. The customer can never pay for that order again; the release_checkout_for_retry / FAILED->SUCCESS / EXPIRED->SUCCESS machinery is dead because no second payment row can ever exist.
- **Verifier trace:** Traced production MoMo path end-to-end. Frontend (ussd-wait.tsx:244) triggers all MoMo pushes via POST /payments/retry; there is no separate first-push endpoint (initiate.py's initiate_checkout_payment is imported only by tests, no router uses it; create_orders_atomic creates no payment row). retry_payment (payment_status.py:386) skips status guards when latest is None, so the FIRST attempt runs _create_retry_payment_attempt with order_id=orders[0].id, producing lenco_reference = ord-<b64(order_id)> (line 270-271). On failure/expiry, sweep_stale_payments (state.py:524-556) sets the row to failed/expired and calls release_checkout_for_retry -> checkout_groups.status='pending'. On retry, latest.status is in RETRYABLE_STATUSES={failed,expired,cancelled} (line 33-39) so the guard passes; retry recomputes the SAME first_order_id (retry even asserts order count unchanged, line 461) and _create_retry_payment_attempt INSERTs a NEW payments row (no delete/update of the old row) with the identical lenco_reference. This violates the payments_lenco_reference_key UNIQUE constraint (0006_money.sql:45). The insert guard at payment_status.py:284 (and initiate.py:107) only handles an empty-data response, not the APIError raised by supabase-py on a PostgREST 409 unique violation, so it surfaces as an unhandled 500. Net effect: after a failed/expired first MoMo payment the customer can never create a second payment row for that order; the release_checkout_for_retry / FAILED->SUCCESS / EXPIRED->SUCCESS machinery is unreachable. The card path (payments_card.py:498) shares the same deterministic-reference + UNIQUE defect. No DB constraint, RLS rule, matrix guard, or exception handler mitigates it. Confirmed real; major severity (order becomes permanently unpayable for the customer).

## 6. 🟡 MAJOR — Reconciliation poller is a poison pill: an illegal transition raises and aborts the whole tick

- **Dimension:** webhooks-state-machines · **Confidence:** high
- **Location:** `services/api/app/services/payments/reconcile.py:379`
- **Problem:** apply_payment_status (state.py:382) checks should_apply_status and, if it returns True, calls transition_payment which raises PaymentTransitionError when the (from_status, event) pair is absent from TRANSITION_TABLE. The table has NO transition out of INITIATED for pay_offline/success/failed (only ussd_pushed and cancelled). But the poller fetches INITIATED payments (NON_TERMINAL_POLL_STATUSES includes PaymentStatus.INITIATED) and, for those, should_apply_status returns True for incoming success (incoming==SUCCESS short-circuits True) and for pay_offline/failed (higher rank than INITIATED). The poll loop's try/except (reconcile.py:366-372) wraps only query_status, NOT apply_payment_status, so the raised AppError propagates out of poll_non_terminal_payments and 500s the entire /internal/reconciliation/poll-tick.
- **Failure scenario:** initiate_checkout_payment inserts a payment as INITIATED, then Lenco's initiate_collection succeeds but the process dies before the ussd_pushed transition (state.py:124). The payment is stuck INITIATED while the customer actually pays; Lenco now reports 'successful'. The poller picks up the INITIATED row (age > 2 min), queries Lenco -> SUCCESS -> should_apply_status True -> transition_payment raises 'Illegal transition initiated + success' -> the whole poll tick 500s. That paid order never reconciles to success (customer charged, order never fulfilled) AND every subsequent poll tick dies on the same poison-pill row, so no other pending payment reconciles either.
- **Verifier trace:** Traced the full path. (1) initiate.py:106 inserts payment status=INITIATED, awaits external initiate_collection (115, where the USSD push reaches the customer), then transitions to USSD_PUSHED at 124 — a crash/deploy/timeout between 115 and 124 leaves the row stuck INITIATED while the customer pays and Lenco reports 'successful'. (2) reconcile.py:25-30 NON_TERMINAL_POLL_STATUSES includes INITIATED; _fetch_non_terminal_payments (330-343) fetches it once older than 2 min, no status exclusion, no ORDER BY. (3) query_status maps 'successful'->SUCCESS (state.py:214-215). (4) should_apply_status(current=INITIATED, incoming=SUCCESS) hits the incoming==SUCCESS short-circuit at state.py:202-203 and returns True (also True for pay_offline rank 20>0 and failed rank 30>0). (5) apply_payment_status (state.py:382-403) calls transition_payment; TRANSITION_TABLE (115-130) has NO INITIATED+SUCCESS/FAILED/PAY_OFFLINE edge, so resolve_transition returns None and transition_payment raises PaymentTransitionError at state.py:337-343 BEFORE the DB update (concurrency guard irrelevant). (6) The raise propagates: apply_payment_status is called at reconcile.py:379, OUTSIDE the try/except that wraps only query_status at 366-372; poll_non_terminal_payments has no other handler; reconciliation_poll_tick (internal_reconciliation.py:41-56) has no handler. Whole tick aborts. (7) The sweeper (_fetch_stale_payments, state.py:453-467) only fetches PAY_OFFLINE/USSD_PUSHED, never INITIATED, so the stuck row is never healed and re-poisons every subsequent tick; the paid order never reconciles and other pending rows are blocked. Only wording correction: PaymentTransitionError.http_status=409, so the endpoint returns 409, not 500 as the finding states — immaterial to the defect. The INITIATED+'pending'->USSD_PUSHED case is the sole legal INITIATED transition, confirming success/failed/pay_offline are all poison variants. Major: money-charged-never-fulfilled plus a self-perpetuating poller outage, gated behind a specific crash/timeout window to seed the row.

- **Location:** `services/api/app/routers/job_completion.py:415`
